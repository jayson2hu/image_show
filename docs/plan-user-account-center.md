# 登录用户个人中心产品方案

日期：2026-05-07  
状态：已完成（2026-05-13 复验通过）  
目标版本：用户侧体验增强第一阶段

## 1. 背景与目标

当前登录用户已经可以生成图片、查看历史记录、查看积分流水、购买套餐，但这些入口分散在顶部菜单和独立页面里。用户登录后缺少一个“我是谁、我还有多少额度、我的作品在哪里、我的账号是否安全、我能修改什么”的集中入口。

本方案目标是新增“个人中心”，让用户登录后可以集中查看和维护自己的账户信息、积分资产、创作资产和通知状态。

业务目标：

- 提升登录后的留存和复访效率。
- 降低用户对积分、有效期、历史作品入口的疑惑。
- 为后续微信登录、账号绑定、订单、公告、用户偏好等能力留出统一承载页。

用户目标：

- 快速知道账号状态、积分余额、积分有效期。
- 快速找到历史作品、积分流水、套餐购买和公告通知。
- 能更新基础个人资料。
- 能看到最近登录、最近生成、最近消费等关键信息。

## 2. 目标用户与核心场景

### 2.1 普通登录用户

用户完成注册或登录后，希望继续生成图片，并确认自己还有多少积分、积分什么时候过期、历史作品是否保存成功。

核心问题：

- 我还有多少积分？
- 我的积分什么时候过期？
- 我之前生成的图片在哪里？
- 我最近一次生成消耗了多少？
- 我的账号邮箱、昵称、头像能不能改？

### 2.2 新注册用户

新用户刚拿到注册赠送积分，需要明确知道注册权益和下一步可以做什么。

核心问题：

- 我注册后获得了什么？
- 这些积分可以生成几张图？
- 免费试用和注册积分有什么区别？
- 生成失败会不会扣积分？

### 2.3 付费用户

用户购买过积分，关心订单、余额、有效期和消费记录。

核心问题：

- 我买的积分到账了吗？
- 我的积分消费到哪里了？
- 失败任务有没有退回积分？
- 我还需要购买哪个套餐？

### 2.4 管理员账号

管理员也是用户，但进入用户侧时不应混淆普通用户体验和后台管理能力。

核心问题：

- 管理员在用户侧可以看到自己的账户信息和积分。
- 后台入口仍然通过 `/console/admin` 进入，不在普通用户中心强展示。

## 3. 信息架构

建议新增路由：`/account`

顶部用户菜单建议调整为：

- 个人中心
- 图片历史
- 积分流水
- 积分套餐
- 退出登录

个人中心页面建议分为 5 个模块：

1. 账户概览
2. 个人资料
3. 积分与权益
4. 创作记录
5. 安全与通知

## 4. MVP 范围

### 4.1 账户概览

展示内容：

- 用户头像。
- 昵称。
- 邮箱。
- 用户身份：普通用户 / 管理员。
- 账号状态：正常 / 禁用。
- 注册时间。
- 最近登录时间。
- 最近登录 IP。
- 当前积分余额。
- 积分有效期。

可用现有字段：

- `users.avatar_url`
- `users.username`
- `users.email`
- `users.role`
- `users.status`
- `users.credits`
- `users.credits_expiry`
- `users.last_login_at`
- `users.last_login_ip`
- `users.created_at`

验收标准：

- 登录用户进入 `/account` 可以看到自己的基础信息。
- 未登录访问 `/account` 跳转登录页。
- 管理员显示“管理员”身份，但不自动跳后台。
- 积分有效期为空时显示“暂无到期时间”或“长期有效”，不得显示空白。

### 4.2 个人资料

第一阶段支持更新：

- 昵称。
- 头像 URL。

暂不支持：

- 修改邮箱。
- 修改密码。
- 上传头像文件。

原因：

- 当前模型已有 `username` 和 `avatar_url` 字段，开发成本低。
- 邮箱和密码涉及验证流程，应放到账号安全阶段。
- 头像文件上传依赖存储策略，建议后续和 R2 文件管理一起做。

验收标准：

- 用户可以修改昵称并保存。
- 用户可以填写头像 URL 并预览。
- 昵称为空时允许保存为空，但页面展示邮箱前缀作为兜底名。
- 头像 URL 不合法时给出友好提示。
- 保存成功后顶部菜单和个人中心同步更新。

### 4.3 积分与权益

展示内容：

- 当前积分余额。
- 积分有效期。
- 最近 5 条积分流水。
- 积分规则说明：
  - 1024x1024 消耗 1 积分。
  - 更大尺寸按像素比例向上取整。
  - 失败或开始前取消按后端规则退回。
- 快捷入口：
  - 查看全部积分流水。
  - 购买积分套餐。

可用现有能力：

- `/api/credits/balance`
- `/api/credits/logs`
- `/packages`

验收标准：

- 用户能在个人中心看到最近积分变动。
- 点击“全部流水”进入 `/credits`。
- 点击“购买积分”进入 `/packages`。
- 积分不足用户能看到明确下一步动作。

### 4.4 创作记录

展示内容：

- 累计生成次数。
- 最近生成时间。
- 最近 6 张作品。
- 最近失败任务摘要。
- 快捷入口：
  - 查看全部历史。
  - 继续生成。

建议后端返回聚合字段：

- `total_generations`
- `completed_generations`
- `failed_generations`
- `latest_generation_at`
- `recent_generations`

第一阶段也可以前端复用 `/api/generations?page=1&pageSize=6`，统计数字先只展示“最近作品”，不做全量聚合。

验收标准：

- 用户能看到最近作品缩略图。
- 点击作品进入历史详情或历史页。
- 没有作品时显示空状态，引导“去生成第一张图”。
- 失败记录显示用户可理解的摘要，不直接展示上游错误原文。

### 4.5 安全与通知

展示内容：

- 最近登录时间。
- 最近登录 IP。
- 登录方式：邮箱 / 微信验证码，后端已有登录日志 `method`。
- 公告未读数量。
- 最近公告。

第一阶段只读展示：

- 最近登录信息。
- 公告入口。

第二阶段再做：

- 修改密码。
- 绑定微信。
- 解绑微信。
- 登录设备列表。
- 安全提醒。

验收标准：

- 用户能看到最近登录时间和 IP。
- 有公告时显示最近公告或未读数量。
- 点击公告可打开现有公告中心。

## 5. 建议新增接口

### 5.1 获取个人中心概览

`GET /api/account/overview`

返回建议：

```json
{
  "user": {
    "id": 1,
    "username": "jayson",
    "email": "user@example.com",
    "avatar_url": "",
    "role": 1,
    "status": 1,
    "credits": 10,
    "credits_expiry": "2026-06-07T00:00:00+08:00",
    "created_at": "2026-05-07T00:00:00+08:00",
    "last_login_at": "2026-05-07T10:00:00+08:00",
    "last_login_ip": "127.0.0.1"
  },
  "credits": {
    "balance": 10,
    "expiry": "2026-06-07T00:00:00+08:00",
    "recent_logs": []
  },
  "creations": {
    "recent_items": [],
    "total": 0,
    "completed": 0,
    "failed": 0,
    "latest_at": null
  },
  "notices": {
    "unread_count": 0,
    "recent_items": []
  }
}
```

说明：

- 该接口只返回当前登录用户自己的数据。
- 不返回 `password_hash`。
- 不返回微信服务端敏感配置。
- 管理员也只能看到自己的用户侧数据，不返回后台系统配置。

### 5.2 更新个人资料

`PUT /api/account/profile`

请求：

```json
{
  "username": "新的昵称",
  "avatar_url": "https://example.com/avatar.png"
}
```

验收标准：

- 只能更新当前登录用户。
- `username` 最长 64 字符。
- `avatar_url` 最长 512 字符。
- `avatar_url` 非空时必须是 `http://` 或 `https://`。
- 返回更新后的用户信息。

## 6. 页面设计建议

页面名称：个人中心

布局建议：

- 顶部：账户卡片，显示头像、昵称、邮箱、身份、注册时间。
- 右侧或顶部指标区：积分余额、有效期、最近登录。
- 中部：两个并列模块。
  - 积分与权益。
  - 最近作品。
- 底部：个人资料编辑、安全与通知。

设计原则：

- 不做复杂设置页，优先做“状态清楚、入口集中”。
- 账户信息不要和后台管理混在一起。
- 积分和历史是最重要的两个用户资产，应优先展示。
- 操作入口要少而明确：编辑资料、查看历史、查看流水、购买积分。

## 7. 功能拆分与开发顺序

### 大功能 1：个人中心只读概览

小功能 1.1：新增 `/account` 路由和入口

- 顶部用户菜单增加“个人中心”。
- 未登录访问跳转登录。

自测：

- 游客访问 `/account` 跳转登录。
- 登录用户菜单能进入个人中心。
- 管理员不会被强制跳后台。

小功能 1.2：个人中心页面骨架

- 展示头像、昵称、邮箱、身份、注册时间、最近登录、积分余额。
- 空字段有兜底文案。

自测：

- 有昵称和无昵称用户显示正常。
- 有头像和无头像用户显示正常。
- `credits_expiry` 为空时显示正常。

小功能 1.3：最近积分流水和最近作品

- 展示最近 5 条积分流水。
- 展示最近 6 张作品。
- 增加入口到 `/credits` 和 `/history`。

自测：

- 无流水、无作品时显示空状态。
- 有数据时点击入口跳转正确。

大功能验收：

- 登录用户能集中看到账号、积分、历史入口。
- 页面不暴露敏感信息。
- 完成后提交：`feat: add account overview`

### 大功能 2：个人资料编辑

小功能 2.1：后端更新资料接口

- 新增 `PUT /api/account/profile`。
- 支持 `username` 和 `avatar_url`。

自测：

- 登录用户可更新自己资料。
- 未登录返回 401。
- 头像 URL 非法返回 400。
- 不允许更新其他用户。

小功能 2.2：前端编辑资料弹窗或内联表单

- 个人中心增加“编辑资料”。
- 支持昵称和头像 URL。
- 保存后刷新用户状态。

自测：

- 修改昵称后顶部菜单同步。
- 修改头像后个人中心预览同步。
- 失败时展示友好错误。

大功能验收：

- 用户可维护基础资料。
- 不影响登录注册流程。
- 完成后提交：`feat: allow users to update profile`

### 大功能 3：账号安全与通知增强

小功能 3.1：最近登录信息增强

- 展示最近登录时间、IP、登录方式。
- 如后端登录日志已有数据，优先复用。

自测：

- 邮箱登录显示邮箱登录。
- 微信验证码登录显示微信验证码。
- 无记录时显示“暂无登录记录”。

小功能 3.2：公告未读与最近公告

- 个人中心展示公告未读数量和最近公告。
- 点击后打开公告中心或跳转公告详情。

自测：

- 新公告显示未读。
- 标记已读后数量减少。

大功能验收：

- 用户能看到安全和通知状态。
- 公告逻辑不影响现有公告中心。
- 完成后提交：`feat: add account security summary`

## 8. 非 MVP 后续能力

暂不建议第一阶段做：

- 修改邮箱。
- 修改密码。
- 绑定/解绑微信。
- 头像文件上传。
- 登录设备管理。
- 订单列表。
- 收藏作品。
- 用户偏好设置。
- 删除账号。

原因：

- 修改邮箱和密码需要更完整的验证码、安全校验和风控策略。
- 微信绑定涉及开放平台或公众号能力，需要确认实际接入模式。
- 订单列表需要支付状态和售后说明一起设计。
- 收藏作品需要新增数据模型，适合和历史资产库第二阶段一起做。

## 9. 数据与隐私要求

- 用户中心只展示当前登录用户自己的数据。
- 不返回密码哈希、微信服务端 token、渠道密钥、R2 密钥等敏感信息。
- 登录 IP 只展示最近一次即可，不展示完整登录历史列表。
- 管理员账号进入用户中心时仍遵循普通用户数据权限。
- 前端不得从后台设置接口读取敏感配置来渲染用户中心。

## 10. 成功指标

上线后观察：

- 登录用户进入个人中心的访问率。
- 从个人中心进入历史页的点击率。
- 从个人中心进入积分流水的点击率。
- 从个人中心进入套餐页的点击率。
- 新注册用户 24 小时内二次生成率。
- 用户因“不知道积分去哪了”产生的问题反馈数量。

## 11. 开放问题

1. 个人中心是否展示订单入口？
   - 建议第二阶段做，先把积分流水和套餐入口打通。
2. 头像是否允许上传文件？
   - 建议后续结合 R2 上传策略统一做。
3. 修改密码是否要第一阶段做？
   - 建议暂缓，等邮箱验证码和微信登录流程稳定后再做。
4. 管理员是否在个人中心显示“进入后台”快捷入口？
   - 建议不显示在普通页面顶部，但可以在个人中心底部用低优先级入口展示，避免普通用户误触。
5. 用户注销账号是否需要支持？
   - 涉及历史作品、订单、积分和合规策略，建议单独立项。

## 12. 推荐 MVP 结论

第一阶段建议只做：

1. `/account` 个人中心入口。
2. 账号概览。
3. 积分余额、有效期、最近流水。
4. 最近作品入口。
5. 昵称和头像 URL 更新。

这组功能能最大程度复用当前已有数据，开发风险低，又能明显改善登录后的用户体验。
