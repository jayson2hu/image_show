# 管理后台站点配置与个人中心体验优化计划

日期：2026-05-08
状态：待开发

## 1. 开发原则

- 每完成一个小功能，先做局部自测和验收，再继续下一个。
- 每完成一个大功能，做整体自测、更新进度文档、提交并推送 GitHub。
- 每修复一个独立问题，单独提交和推送，提交信息写清楚改动范围。
- 发现需求矛盾、支付渠道口径、权限边界不清时，先记录问题并询问确认。
- 不把管理员能力和普通用户能力混在同一个入口里；管理员也是用户，但管理后台只处理运营配置。

## 2. 范围拆分

### 大功能 A：管理后台站点配置

目标：让管理员可以配置站点基础信息、SEO 信息、注册策略和站点关于内容，减少代码写死。

#### A1. 后端设置键和默认值梳理

开发内容：
- 梳理现有 `settings` 表和 `/api/admin/settings` 接口。
- 新增站点基础设置键：
  - `site_title`
  - `site_about`
  - `seo_title`
  - `seo_keywords`
  - `seo_description`
  - `register_enabled`
  - `register_email_domain_allowlist`
- 增加默认值，保证老数据库启动后不报错。

自测：
- `go test ./controller -run "TestAdmin.*Setting|Test.*Setting" -v`
- 直接请求 `GET /api/admin/settings`，确认新字段存在。

验收标准：
- 老数据兼容。
- 空配置时前端有默认展示。
- 管理员保存后刷新仍能读取。

进度记录：
- 完成后更新 `docs/progress.md`。
- 提交并推送：`feat(admin): add site seo settings backend`

开发内容：

验收标准：

- 停用套餐不在前台展示。

#### B3. 充值渠道配置

### 大功能 D：个人中心资料与头像上传

目标：个人中心只展示和编辑当前用户自己的信息，头像支持本地上传，更新后所有使用头像的地方同步。

#### D1. 头像上传后端接口

开发内容：
- 新增 `POST /api/account/avatar`。
- 仅允许登录用户上传自己的头像。
- 限制文件类型：jpg、jpeg、png、webp。
- 限制文件大小，建议第一版 2MB。
- 第一版头像不上传到 R2，先使用本地头像存储。
- 后台增加头像存储配置：
  - `avatar_storage_driver`，第一版默认 `local`。
  - `avatar_max_size_mb`，默认 2。
  - `avatar_allowed_types`，默认 `jpg,jpeg,png,webp`。
- 本地头像保存到后端可公开访问的头像目录，并生成稳定访问 URL。
- 更新当前用户 `avatar_url`。

自测：
- `go test ./controller -run "TestAccount.*Avatar|TestAccount" -v`
- 覆盖未登录、过大文件、非法类型、正常上传。

验收标准：
- 不能上传脚本或超大文件。
- 更新后 `/api/auth/me` 和 `/api/account/overview` 返回新头像。
- 不影响管理员角色。
- 后台修改头像大小限制后，上传校验按新配置生效。

进度记录：
- 完成后更新 `docs/progress.md`。
- 提交并推送：`feat(account): support avatar upload`

决策记录：
- 第一版头像不上传 R2，使用后台可配置的本地头像存储。
- 后续如启用 R2，需要增加后台迁移能力，统一上传已有用户头像并回写头像地址。

#### D2. 后台头像存储配置 UI

开发内容：
- 在后台设置页增加“头像存储”分组或放入“图片存储”分组。

  - 头像最大大小 MB。
  - 允许的头像格式。
- 增加说明：当前头像不走 R2，后续如开启 R2 需要先执行迁移。

自测：

验收标准：

## 5. 风险和待确认

1. SEO 配置对 SPA 的限制：
   - 前端运行时修改 meta 对搜索引擎抓取能力有限。
   - 如果后续要求强 SEO，需要考虑服务端渲染或静态预渲染。
2. 真实支付渠道：
   - 当前已确认第一版只做人工充值联系方式。
   - 后续如接真实支付，需要另起支付集成计划。
3. 头像 R2 迁移：
   - 当前已确认第一版不上传 R2。
   - 后续启用 R2 前，需要实现后台一键迁移，并更新全部用户头像地址。
